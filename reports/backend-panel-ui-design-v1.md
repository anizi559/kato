# 自研代理面板后台 UI 设计 v1

记录日期：2026-06-15

## 0. 设计目标

这份文档定义 Backend Core 管理后台的第一版 UI 信息架构和页面设计。目标不是做漂亮展示页，而是做一个高密度、低误操作、可快速排障的运维控制台。

后台 UI 只通过 Frontend Edge 暴露给管理员。Backend Core 不直接暴露给浏览器。

第一版 UI 必须服务这些核心工作：

- 管用户、套餐、权限。
- 管代理节点、协议入站、用户可见访问节点。
- 管中转服务器、转发规则、Access Node 联动。
- 管 Frontend Edge、Subscription Edge、Agent 注册和状态。
- 发布配置、查看版本、回滚配置。
- 查看健康状态、告警、日志和审计。
- 在 Backend Core 或 Agent 异常时快速判断影响范围。

## 1. 设计原则

### 1.1 运维控制台优先

- 第一屏是工作台，不是营销页。
- 默认布局偏紧凑，表格信息密度高，但保留清晰行高和状态标签。
- 重要信息优先展示状态、版本、影响范围、最近错误、下一步操作。
- 删除、吊销、回滚、重置凭据、发布配置等高风险操作必须二次确认。

### 1.2 分离架构必须可见

UI 里不要把“节点”做成一个笼统概念。必须明确区分：

- `Proxy Node`：真实落地代理服务器。
- `Node Inbound`：落地节点上的协议入站。
- `Transit Relay`：真实中转服务器。
- `Relay Rule`：中转服务器上的转发规则。
- `Access Node`：用户订阅里看到的可访问节点，可以是直连，也可以是中转。
- `Frontend Edge`：管理员/用户访问入口。
- `Subscription Edge`：公开订阅入口。

### 1.3 用户视角和运维视角分离

管理员配置的是 Proxy Node、Transit Relay、Node Inbound、Relay Rule。

用户最终看到的是 Access Node。Access Node 页面应该成为“订阅输出视角”的中心页面。

### 1.4 默认保守

- 第一版只做单 super admin + 基础权限预留。
- 不做代理商角色。
- 不做支付、订单、价格。
- 不做复杂多级中转链。
- 线路测试、质量评分放在 P1，不进入第一版核心 UI。

## 2. 全局框架

### 2.1 布局

采用左侧主导航 + 顶部状态栏 + 主内容区。

顶部状态栏：

- 当前系统版本。
- 当前配置版本。
- Backend Core 状态。
- 未处理告警数量。
- 最近一次配置发布状态。
- 当前管理员菜单。

主内容区顶部统一结构：

- 页面标题。
- 一句话说明当前页面管理什么资源。
- 主要动作按钮。
- 次要动作菜单。
- 最近同步时间或数据刷新状态。

### 2.2 全局交互

全局需要支持：

- 表格搜索。
- 状态筛选。
- 标签筛选。
- 分组筛选。
- 批量选择。
- 列显示配置。
- 复制 ID / 复制命令 / 复制订阅链接。
- 一键跳转关联资源。
- 危险操作二次确认。
- 操作结果 toast。
- 长任务抽屉显示进度。

### 2.3 状态标签

统一状态语义：

- `正常`：资源可用，无错误。
- `停用`：管理员主动停用。
- `待注册`：资源已创建，但 Agent/Edge 未注册。
- `在线`：最近心跳正常。
- `离线`：超过心跳超时。
- `降级`：可用但存在告警。
- `故障`：影响服务。
- `待发布`：配置已变更但未下发或未确认应用。
- `发布中`：正在下发配置。
- `已应用`：Agent 已确认配置版本。
- `回滚中`：正在回滚配置。
- `离线容灾`：Backend Core 不可达，Agent 使用 last known good。

## 5. 总览 Dashboard

### 5.1 页面目标

管理员打开后台后，30 秒内回答：

- 当前服务是否可用。
- 哪些 Agent 离线。
- 最近一次配置发布是否成功。
- 用户是否有大面积过期/超流量。
- 节点或中转是否有故障。

### 5.2 布局

第一行：系统关键状态。

- Backend Core：健康 / 降级 / 故障。
- 配置版本：当前版本、待发布数量。
- Agent：在线数 / 总数。
- 告警：严重 / 警告。
- 今日流量。
- 本月流量。

第二行：资源摘要。

- 用户：正常、停用、过期、超流量。
- 访问节点：正常、停用、故障。
- 代理节点：在线、离线、故障。
- 中转服务器：在线、离线、故障。

第三行：操作队列。

- 最近配置发布。
- 最近 Agent 上报。
- 最近告警。
- 最近审计日志。

### 5.3 主要动作

- 发布配置。
- 查看待处理告警。
- 创建用户。
- 添加代理节点。
- 添加中转服务器。

### 快速创建中转访问节点流程

入口位置：

- Node Inbound 行内按钮。
- Proxy Node 详情页的入站标签页。
- Access Node 列表顶部按钮。

步骤：

1. 选择源 Node Inbound。
2. 选择 Transit Relay。
3. 设置入口端口。
4. 设置 TCP/UDP。
5. 设置显示名称、标签、分组。
6. 预览生成结果。
7. 确认创建。

确认后自动生成：

- `relay Access Node`。
- `Relay Rule`。

创建成功页显示：

- 用户订阅入口地址。
- 中转规则 `entry -> target`。
- 需要发布到哪些 Agent。

## 14. 核心工作流

## 14.1 添加 VLESS REALITY 直连节点

1. 进入代理节点。
2. 创建 Proxy Node。
3. 生成 Agent 接入命令。
4. 节点 Agent 注册后状态变为在线。
5. 创建 Node Inbound，协议选 VLESS REALITY。
6. 系统自动创建 direct Access Node。
7. 发布配置。
8. 在访问节点页确认用户可见入口。

## 14.2 给已有入站添加中转入口

1. 在协议入站行点击“创建中转访问节点”。
2. 选择 Transit Relay。
3. 输入入口端口和传输类型。
4. 预览 Access Node + Relay Rule。
5. 确认创建。
6. 发布配置。
7. 在访问节点和转发规则两个页面都能看到联动结果。

## 14.3 添加 Hysteria2 节点

1. 创建或选择 Proxy Node。
2. 创建 Hysteria2 Node Inbound。
3. 选择证书引用。
4. 开启 obfs 并生成独立 obfs password。
5. 设置 up/down Mbps。
6. 自动创建 direct Access Node。
7. 可选创建中转 Access Node。
8. 发布配置。

## 14.4 Agent 离线容灾排障

1. 总览出现 Agent 离线或离线容灾标签。
2. 进入 Agent 详情。
3. 查看最近 actual state、last known good、配置版本。
4. 跳转 Proxy Node 或 Transit Relay。
5. 查看进程/端口状态。
6. 如果当前服务仍运行，标记为降级而不是故障。
7. Backend Core 恢复后确认 Agent 自动同步并补报状态。

- 大屏桌面：保持云控制台布局，左侧导航 + 主资源表格 + 右侧详情面板。
- 窄桌面 / 浏览器缩放：左侧导航压缩为图标栏，详情面板移动到表格下方，避免表格和详情互相挤压。

资源页模板规则：

- P0 资源列表页优先复用同一套模板：页面标题区、搜索与筛选、分组表格、分页、右侧详情面板。
- 用户、套餐、访问节点、代理节点、协议入站、中转服务器、转发规则、Agent 都应该使用同一套响应式资源页骨架。
- 每个资源页只替换数据字段、筛选项、主操作按钮和详情面板内容，不重新定义页面结构。
- 右侧详情面板在桌面固定在表格右侧，在窄桌面和缩放场景移动到表格下方，在手机保持全宽顺序阅读。
- 表格字段要声明稳定列宽；手机端自动转换为带字段标签的资源卡片。
- 当前选中项必须跟随筛选后的数据集，避免详情面板显示已经被筛掉的资源。

## 17. 已确认的设计决策

确认时间：2026-06-15。以下问题按建议方案锁定，用作后续前端路由、页面草图和组件设计的依据。

1. 访问节点作为主导航一级页面。它代表用户最终订阅视角，也是直连入口和中转入口的统一展示对象。
2. 代理节点详情页使用 tabs 展示协议入站、访问节点、Agent、配置、流量和日志，避免详情页向下堆叠过长。
3. VLESS REALITY 和 Hysteria2 放在同一个“协议入站”页面，表单根据协议类型切换字段。
4. 第一版配置发布采用手动发布按钮。资源保存后只进入“待发布”状态，由管理员显式发布，降低误操作风险。
5. Frontend Edge 本地设置后台后续单独设计，不混在 Backend Core 管理后台里。Backend Core 只展示 Frontend Edge 的注册、在线状态、版本、域名和摘要信息。
